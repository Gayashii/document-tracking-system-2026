# Task 2.5 — RBAC Middleware and Role-Scoped Data Access

**Phase:** 2 — Core Features
**Status:** [x] Complete

## Objective
Enforce role-based access control at the route level and ownership-based access at the service level across all endpoints.

## Steps

### Route-Level RBAC (`middleware/authorize.js`)
- [x] `authorize(allowedRoles[])`: factory that returns middleware checking `req.user.role`
- [x] On failure: 403 `{ success: false, error: { code: 'FORBIDDEN', message: 'Insufficient role' } }`
- [x] Apply to all existing and new routes

### Service-Level Ownership Guards
- [x] `documentService.getById(documentId, actor)`:
  - If actor role is `student`: only return document if `documents.student_id === actor.id`
  - If actor role is `auditor`: return document (read-only access)
  - If actor role is `finance_staff` or `admin`: return any document
  - Throw `ForbiddenError` if student tries to access another student's document
- [x] `documentService.list(filters, actor)`:
  - Automatically append `WHERE student_id = actor.id` for student role
  - Finance staff and admin see all documents
  - Auditor sees all documents (read-only)

### Role Permission Matrix (document in `constants/roles.js`)
| Action | Admin | Finance Staff | Student | Auditor |
|---|---|---|---|---|
| Upload document | ✓ | ✓ | ✓ (own) | ✗ |
| View any document | ✓ | ✓ | own only | ✓ (read) |
| Change document status | ✓ | ✓ | rejected→submitted | ✗ |
| Delete document (soft) | ✓ | ✗ | ✗ | ✗ |
| View audit log | ✓ | ✗ | ✗ | ✓ |
| Manage users | ✓ | ✗ | ✗ | ✗ |
| Export reports | ✓ | ✓ (dept) | ✗ | ✓ |

### Admin-Only Soft Delete
- [x] `DELETE /api/v1/documents/:id` — sets `documents.deleted_at` (soft delete; file on disk is retained)
- [x] Deleted documents excluded from all list queries by default
- [x] Deletion logged to `audit_logs` with actor, timestamp, IP

### Tests
- [x] Unit: `authorize(['admin'])` blocks student and finance_staff roles
- [x] Integration: student requesting another student's document returns 403
- [x] Integration: auditor can GET document but cannot PATCH status (returns 403)
- [x] Integration: only admin can soft-delete

## Acceptance Criteria
- Every API endpoint has explicit role guards
- Students cannot access other students' documents under any circumstance
- Auditors have read-only access with no way to trigger write operations
- All access denials are logged in `audit_logs`
