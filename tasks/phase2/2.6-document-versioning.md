## Objective
Allow students to re-submit a new file version after rejection, archiving the previous version while preserving the document record and history.

## Steps

### Service (`documentService.js` addition)
- [x] `createVersion(documentId, file, actorId, changeNote)`:
  1. Verify document status is `rejected` and actor is the document owner (student)
  2. Archive current `file_path` by inserting to `document_versions` (with current version number)
  3. Save new file to `server/uploads/documents/{referenceNumber}/v{newVersion}/` via storage service
  4. In a single DB transaction:
     - Update `documents.file_path` to new path
     - Increment `documents.version`
     - Reset `documents.status` to `submitted`
     - Insert `document_status_history` row (rejected â†’ submitted, note: "Re-submission v{n}")
     - Insert `document_versions` row archiving the old file
     - Insert `audit_logs` row
  5. Emit `document.status_changed` event (triggers notification to finance staff)

## Acceptance Criteria
- Students can re-submit only after a rejection (not from any other status)
- Previous file versions remain accessible and downloadable
- Finance staff sees the version number and full history on the document detail page
- Each version change is logged in `audit_logs`
