# Task 4.6 — UAT and Go-Live

**Phase:** 4 — Production Readiness
**Status:** [ ] Not Started

## Objective
Coordinate user acceptance testing, parallel running with the old manual process, and final production cutover.

## Steps

### Production Data Setup
- [ ] Run `knex migrate:latest` on production database
- [ ] Seed production data (no dev seed data):
  - Admin user (strong password, MFA enabled)
  - All departments from the Postgraduate Institute Financial Section
  - All document types currently in use
- [ ] Verify seed ran correctly: log in as admin, confirm departments and document types exist

### UAT Environment
- [ ] Deploy the staging environment to a URL accessible by Finance Staff testers
- [ ] Create test accounts: 2 Finance Staff, 2 Student, 1 Auditor
- [ ] Provide UAT test script covering:
  - Student: register, log in, upload each document type, view status, respond to rejection
  - Finance Staff: log in, view pending queue, approve document, reject document with note, assign barcode number to document
  - Admin: create user, deactivate user, change system settings, view audit log, export report
  - Auditor: log in, view documents, view audit trail, download report

### UAT Sign-Off
- [ ] Finance Staff sign-off: all document workflows tested and approved
- [ ] Head of Finance sign-off: approval/rejection workflow confirmed
- [ ] Admin sign-off: user management and system settings confirmed
- [ ] Auditor sign-off: audit trail and export confirmed
- [ ] Document any bugs found during UAT → fix → retest before go-live

### Pre Go-Live Checklist
- [ ] `npm audit` — zero high-severity vulnerabilities
- [ ] OWASP ZAP scan on staging — zero high/medium severity findings
- [ ] SSL certificate installed and HSTS enabled
- [ ] MFA enforced for admin and finance_staff roles
- [ ] Uptime monitoring configured and alerting to correct contacts
- [ ] All smoke tests pass on production URL
- [ ] Email deliverability tested: send test notification to student email, verify arrives in inbox (not spam)
- [ ] SPF, DKIM, DMARC DNS records configured for sending domain

### Parallel Running Period (2 weeks)
- [ ] Run new system alongside old manual process for 2 weeks
- [ ] Finance staff processes documents in both systems simultaneously
- [ ] Daily sync check: verify document counts match between systems
- [ ] Any discrepancy investigated immediately
- [ ] Document outcome of parallel running; get sign-off from Head of Finance before cutover

### Production Cutover
- [ ] Announce cutover date to all users (email + in-system notice)
- [ ] On cutover day:
  1. Confirm production health check is green
  3. Switch DNS / update URL if needed
  4. Send "System is live" notification to all users
  5. Finance Staff stop using old manual process
- [ ] Monitor error logs closely for 48 hours post-cutover

### Post Go-Live
- [ ] Schedule first security review in 3 months
- [ ] Collect user feedback after 2 weeks of live operation
- [ ] Create backlog of Phase 2 future enhancements (AI chatbot, payment gateway integration)

## Acceptance Criteria
- All four user roles have completed UAT and signed off
- Zero high/medium security findings from OWASP ZAP
- Parallel running period completed with zero discrepancies
- Production system has been live for 48 hours with zero critical errors
- All users have received credentials and training

## Dependencies
- All Phase 1–4 tasks complete
